# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A personal Neovim configuration, plugin-managed via lazy.nvim. This is not an
application with a build/test/run cycle — "changing the code" means editing
Lua config files that take effect the next time Neovim starts (or via
`:source %` / `:Lazy reload <plugin>` during a session).

## Structure

- `init.lua` — entry point. Sets leader keys, disables netrw (snacks.nvim's
  explorer replaces it), then requires the three modules below in order.
- `lua/config/options.lua` — core `vim.opt` settings (indentation, search,
  Treesitter-based folding).
- `lua/config/keymaps.lua` — global keymaps and user commands not tied to a
  specific plugin.
- `lua/config/lazy.lua` — bootstraps lazy.nvim (clones it if missing) and
  calls `require("lazy").setup("plugins", ...)`, which auto-loads every spec
  file under `lua/plugins/`.
- `lua/plugins/*.lua` — one file per concern, each returning a lazy.nvim spec
  table (a list of plugin specs). Add a new plugin by adding an entry to the
  relevant file, or a new file if it's a new concern — no registration step
  needed beyond that, lazy.nvim picks up any file in `lua/plugins/`.
- `lazy-lock.json` — pinned plugin commits, generated by lazy.nvim. Don't
  hand-edit; regenerate via `:Lazy` operations.

## Working with plugin specs

- Each file in `lua/plugins/` returns a flat list of plugin spec tables
  (lazy.nvim format: `{ "author/repo", opts = {...}, keys = {...}, ... }`).
  Group by concern (LSP, completion, git, UI, etc.), not one-file-per-plugin.
- Prefer `opts = {}` over a `config` function unless setup requires logic
  (e.g. `lsp.lua` and `treesitter.lua` use `config`/`init` because they
  register autocmds and dynamically compute installed parsers).
- LSP servers are configured with the native `vim.lsp.config()` /
  `vim.lsp.enable()` API (Neovim 0.11+), not `lspconfig`'s old `.setup{}`
  wrapper — see `lua/plugins/lsp.lua`. `mason-lspconfig`'s `ensure_installed`
  only installs binaries; the servers still must be registered and enabled
  explicitly in the `nvim-lspconfig` spec's `config` function.
- `rustaceanvim` intentionally owns Rust LSP attach instead of `rust_analyzer`
  via mason-lspconfig/nvim-lspconfig — don't add `rust_analyzer` to the
  `ensure_installed`/`enable` lists in `lsp.lua`.
- When two plugins could plausibly own the same job, the comments in that
  spec file explain the division of labor — read them before changing
  behavior. Notable ones:
  - `snacks.nvim`'s explorer (`ui.lua`) replaces netrw and owns
    directory-opening (VSCode-sidebar model); `oil.nvim` (`explorer.lua`)
    stays available on-demand via `:Oil` for edit-directory-as-buffer /
    bulk-rename.
  - `flash.nvim` is for jumping to any visible on-screen location; it
    complements `fzf-lua` (project-wide file/text search) and `aerial.nvim`
    (in-buffer symbol navigation) rather than replacing them.

## Commands (run from inside Neovim)

- `:Lazy` — open the plugin manager UI (install/update/clean/profile).
- `:Lazy sync` — install missing plugins and update to latest per
  `lazy-lock.json` constraints.
- `:Mason` — manage LSP server / formatter / linter binaries.
- `:checkhealth` — diagnose Neovim/plugin setup issues.
- `:Term` — custom command (in `keymaps.lua`) that opens a terminal in a
  split.

## Conventions

- Formatting and linting are handled by `conform.nvim` and `nvim-lint`
  (`lua/plugins/format-lint.lua`), not by editor-agnostic project tooling —
  there's no repo-level `package.json`/`Makefile` to invoke.
- Leader is `<space>`. Check `lua/config/keymaps.lua` and the `keys` table in
  each plugin spec before adding a new mapping, to avoid collisions.
